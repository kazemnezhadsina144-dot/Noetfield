Purpose

Procurement reviews stall when “deliverables” are ambiguous. This specification defines concrete artefacts, evidence standards, acceptance gates, and a review workflow so Security, Legal, IT, and Procurement evaluate the same objects.

Contract truth (simple)

Acceptance is defined by versioned deliverables + explicit assumptions/gaps—never by undefined “completion.” If evidence access is constrained, deliverables still ship and constraints are recorded as gaps.

Deliverables (versioned artefacts)

Default pack (board-ready). Add-ons are scoped explicitly.

Board memo (PDF)

Decision-ready memo that stands alone.

  • Decision statement + rationale
  • Options and tradeoffs
  • Actions, owners (roles), timelines
  • Scope, exclusions, assumptions, gaps

Trust Ledger entry (HTML)

Versioned record of system/use-case, controls, and evidence pointers.

  • System/use-case identifiers
  • RACI ownership (by role)
  • Controls mapped to evidence
  • Gaps and follow-up tasks

Controls map (PDF/HTML)

Traceability from risks → controls → proof.

  • Risk categories + impact notes
  • Control definitions + owners
  • Proof pointers (tiers)
  • Remediation actions (prioritized)

Evidence index (PDF)

Reviewed vs missing vs assumed—explicit.

  • Reviewed evidence list
  • Missing evidence list
  • Assumptions register
  • Access constraints log
Optional add-ons (scoped) enterprise
  • Vendor questionnaire completion (SIG Lite / CAIQ) with evidence references
  • Policy-to-control mapping exhibits (board + audit alignment)
  • Executive risk register slice (per system or program)
  • Evidence request list (internal data collection plan)
RACI (roles and ownership)

Ownership is specified by role (not person). Buyer can map roles to internal titles.

Artefact / Decision Accountable (A) Responsible (R) Consulted (C) Informed (I)
Board memo decision statement Executive sponsor Program owner Security, Legal/Privacy, IT Procurement, Audit (as required)
Controls map approval Security owner IT owner Risk/Compliance, Program owner Executive sponsor
Evidence index sign-off Risk/Compliance Program owner Security, IT Executive sponsor, Audit
Trust Ledger entry publication Program owner Noetfield (delivery lead) Security, IT Stakeholders (as scoped)
RACI is an output, not a negotiation. Buyer provides role mapping once at kickoff; the pack ships with role ownership labels.
Evidence standards (proof tiers)

Evidence pointers are explicit. Where evidence is unavailable, the gap is recorded and tied to an owner and due date.

Tier Name Examples Use
P0 Statement only Verbal description; undocumented control Accepted only as a gap; requires follow-up proof
P1 Documentary Policy, SOP, configuration note, change record Baseline evidence where system access is limited
P2 System evidence Admin screenshots, audit logs, settings export Preferred proof for control operation
P3 Independent validation 3rd-party audit attestation; formal test report Highest confidence for critical controls

Non-negotiable rule

No control is described as “in place” without a stated proof tier. If proof is P0, it is listed as a gap.

Acceptance gates (A0–A3)

A simple enterprise-friendly workflow: align → draft → review → final acceptance.

Gate Meaning Inputs required Outputs produced Acceptance check
A0 Scope lock Use-case list, role mapping, constraints Scope + exclusions + evidence plan Scope is signed off by accountable role
A1 Draft pack Available evidence (P1–P3) + interviews Draft memo + draft ledger + draft evidence index All required sections present; gaps enumerated
A2 Review & redlines Reviewer comments (Security/Legal/IT) Redline log + updated drafts Each redline is accepted/rejected with rationale
A3 Final acceptance Final reviewer sign-off Versioned final pack + change notes Deliverables match the spec; version stamped
If evidence access is constrained, acceptance still proceeds provided gaps and access constraints are recorded explicitly and tied to owners.
Review workflow (enterprise-friendly)

A predictable review cadence that reduces meeting load and prevents scope drift.

Reviewer roles

  • Security: controls + evidence tiers
  • Legal/Privacy: data boundaries, DPA alignment
  • IT: feasibility, system constraints
  • Procurement: SOW acceptance and invoicing

Redline discipline

  • All comments captured in a redline log
  • Each comment resolved with rationale
  • Changes roll into a versioned revision
  • No silent edits; change notes included

Meeting minimization (default)

Two required sessions: kickoff and final review. Everything else is async via redline log, unless buyer requests additional sessions.

Real scenario example (condensed)

Example scenario shows what a board-ready pack looks like. All names are illustrative; the structure is the product.

Scenario

A 1,200-person regulated services firm plans to deploy Microsoft 365 Copilot to 300 users (Sales, Customer Support, Operations). Key concern: oversharing exposure from SharePoint/Teams permissions + sensitive client data in email. Procurement requires a deliverable-defined engagement, and Security requires proof tiers and an evidence index.

Scope (A0)

  • In scope: M365 tenant posture relevant to Copilot, data boundaries, access controls, retention basics
  • Out of scope: full security audit, pentest, legal advice, production data extraction
  • Constraints: read-only access; evidence via exports/screens; no credentials via email

Decision

  • Board decision: proceed with phased enablement (pilot → controlled expansion)
  • Condition: close 6 identified high-risk oversharing gaps before expansion
  • Owners: roles mapped in RACI
Example: Board memo (condensed) PDF artefact
Section Content (example)
Decision Approve Copilot pilot (60 users) with expansion gated on evidence-backed controls for oversharing, retention, and sensitive-data boundaries.
Options Option A: Proceed now with gating controls (recommended).
Option B: Delay 8–12 weeks for broader remediation.
Option C: Proceed without gating (not recommended).
Top risks Oversharing from permissive SharePoint/Teams access; sensitive data in mailboxes; retention gaps; weak exception handling for connectors.
Actions 6 actions with owners and due dates (security owner + IT owner + program owner), tied to evidence tiers.
Assumptions Assumption register included; any P0 controls listed as gaps with owners and due dates.
